# Operator UI Product Review

## Purpose

Review an operator-facing interface like a control surface, not like a
marketing page.

## Workflow

1. Identify the real user and action model.
   Determine who uses the interface, what they are trying to do, what must stay
   visible, and what should remain impossible from that surface.
2. Check information hierarchy first.
   Verify that status, scope, risk, action boundaries, and next steps are more
   obvious than decorative chrome.
3. Review accessibility and interaction safety.
   Check keyboard access, contrast, focus visibility, labels, error recovery,
   and whether risky actions are explicit.
4. Review data-boundary honesty.
   Confirm the UI makes derived-versus-source-data boundaries visible and does
   not imply authority it does not have.
5. Record findings in operator terms.
   Prefer clarity, safety, and workflow comprehension over generic design
   critique.

## Focus Areas

- system status visibility
- action classification and approval clarity
- dense but legible information architecture
- case board and workspace orientation
- route contract versus router or smoke drift
- review-only versus control-surface honesty
- derived-output versus source-data boundary signals
- repeated boundary readouts versus scanability
- keyboard and accessibility quality

## Do Not Use It For

- brand-polish critique without workflow evidence
- generic frontend review that ignores operator risk
- visual novelty recommendations that weaken comprehension
